Nginx-整理

发表于2022-10-23|更新于2025-08-10|linuxnginx
|阅读量:

基本概念

​ nginx 是一个开源的，支持高性能，高并发的 www 服务和代理服务软件。它是俄罗斯人 lgor sysoev 使用 C 语言开发的，nginx 占用的系统资源更少，使用基于事件驱动的架构，单节点能支持万级别的并发连接

​ nginx 不但是一个优秀的 web 服务软件，还可以作为反向代理，负载均衡，以及缓存服务使用

工作原理

​ 一般情况下，启动 nginx 服务时会有一个 master 进程和几个 worker 进程（通过配置中设定），它们之间通过事件信号来通信。对于 worker 进程会根据配置把请求给到后面的应用服务器，当请求转发后，会继续处理后续的请求，当应用服务器返回响应时，会收到事件通知来处理响应

​ 所以 nginx 的逻辑是多进程 + 异步 IO 多路复用实现的高并发

安装

​ 可以在官方网站下载安装包，wget -c https://nginx.org/download/nginx-1.16.1.tar.gz，这样下载后的文件是源码文件是需要进行遍历安装的

​ 通常情况下，在正式安装之前，还需要安装一些相关的依赖 yum install gcc zlib-devel bzip2-devel pcre pcre-devel openssl openssl-devel -y

​ 安装依赖完成后，对 nginx 的安装包进行解压，tar -zxvf nginx-1.16.1.tar.gz，然后进入到解压后的目录中，开始释放 makefile 文件 ./configure --prefix=/usr/local/niginx1.16，prefix 参数指定最后 nginx 的安装位置

​ 然后开始编译安装 make && make install，安装完成后，进入安装目录，使用 nginx 就可以启动 nginx 服务，如果要指定启动的 nginx 配置文件，可以使用 nginx -c 去指定配置文件的目录，如果要关闭 nginx 可以使用 nginx -s stop，如果要重载配置文件可使用 nginx -s reload

​ 除此之外，一种更便捷的方式就是使用 docker 容器，安装好 docker 后，使用 docker pull nginx 就是下载最新的 nginx 镜像

​ 然后通过镜像文件运行 nginx 容器 docker run --name nginx-test -p 8080:80 -d nginx，–name 指定容器的名称，-p 将本机的 8080 端口映射到 nginx 容器的 80 端口，-d 表示让容器在后台运行，最后的 nginx 其实就是镜像文件的名称

​ 容器启动后，本地访问 8080 就会将请求发送到 nginx 容器

​ Ubuntu 上也可以更快捷使用 sudo apt-get install nginx 进行安装，然后使用 sudo systemctl nginx start 启动

配置

​ 安装完成后，在 /etc/nginx 目录下会有以下目录 conf.d fastcgi_params mime.types modules nginx.conf scgi_params uwsgi_params

​ nginx.conf 是 nginx 服务的配置文件，里面的 include /etc/nginx/conf.d/*.conf 就是让 nginx 去加载对应的配置

​ conf.d 目录里存放这不同网站的配置文件，里面会有一个默认的配置文件 default.conf，这个文件其实就是一个参考文件，实际运用中该文件可以删除

​ 对于配置文件来说，可以分成三块：

​ 全局块，即上面横线隔开的第一部分，用于定义进程数，日志的路径等

​ events 块，用来配置事件处理模型，进程最大的连接数，以及 http 协议的 keep-alive 的超时时间

​ http 块，用来配置路由，设置静态文件代理，反向代理，负载均衡等

​ 注意，其实 http 块内部还可以分成三部分：首先是 http 块内部第一部分 全部块，这个配置会对所有 nginx 的站点生效；server 块，是针对单站点的配置；location 块，是针对单个页面或是 url 配置，以及静态文件下载等

​ 对于 location 的路由配置，支持普通字符串匹配和正则匹配，正则匹配中各种不同方式有着不同的优先级

匹配符 匹配规则 优先级
= 精确匹配 1
^~ 以某个字符串开头 2
~ 区分大小写的正则匹配 3
~* 不区分大小写的正则匹配 4
!~ 区分大小写的不匹配正则 5
!~* 不区分大小写的不匹配正则 6
/ 通用匹配，任何请求都会匹配到 7

​ 安装完 nginx 后，在 /etc/nginx/ 目录下可能还会有两个目录，即 sites-available 和 sites-enabled，前者通常就是针对多个不同站点，将各自的配置文件放在该文件夹下，后者则是一个软链地址，软链的目标就是 sites-available 里的配置文件

​ sites-available 表示可以用来存放可以使用的站点配置文件，sites-enabled 表示启用的站点配置文件，也就是说当站点暂时不使用时，可以在 sites-enabled 将对应配置文件删除，但实际的配置文件还在 sites-available 中，所以如果后期要恢复，只需要再创建软链即可

​ 创建软链使用 ln -s 源文件 目标文件

代理

​ nginx 的代理分为了两种，一种是正向代理，另一种是反向代理。所谓代理，其实就是中间者，一个渠道，所以对于代理来说，就需要 发起者 和 接收者，前者就是客户端，后者自然就是提供服务的服务器

正向代理

​ 其实就是将 nginx 和 客户端 划分在一起作为一个整体，不同的客户端将请求发送到 nginx 服务器，通过这个服务器作为客户端的代理，将请求按照配置发送到指定的处理服务的服务器地址

反向代理

​ 顾名思义，以上面正好相反，反向代理可以将 nginx 和 服务处理服务器 作为一个整体，那么对于客户端来说，自己其实并不知道实际提供服务处理的服务器，也就是说客户端相当于是在和 nginx 做交互

​ 实现反向代理的配置，在 server 块的 location 中配置

负载均衡

​ 通常随着项目的请求逐步增加，考虑到项目的健壮性，以及扩展到分布式的架构等，用于处理请求的服务器会增多，那么方向代理时，就需要通过负载均衡，来对请求进行分发

​ nginx 默认情况下，是按照请求的时间逐一分发到目标服务器，也就是 默认是按照轮询方式

​ 除此之外，nginx 的负载均衡 还支持 weight 权重方式，ip_hash ip 哈希分发方式，url_hash url 哈希分发方式，fair 响应时间分发方式

限流

​ 限流（Rate Limitting）是服务降级的一种方式，常见的限流算法有：漏桶算法，令牌桶算法

​ 漏桶算法，就是请求会进入一个设定的匀速出水的漏桶中，当请求过多装满桶时，后续请求就被拒绝，对应的数据结构就是一个 FIFO 的队列。在 nginx 中配置限流使用的是漏桶算法

​ 令牌桶算法，就是请求会进入队列中，所有要被处理的请求需要从令牌桶里获取到令牌才可以被处理，如果队列满了，或桶里没有令牌时请求就被拒绝。令牌桶通过设定桶容量和令牌发放的速度来控制流量限制

​ 但是直接丢弃请求或拒绝会严重影响访问，所以上面这个两个算法都允许了对应突然处理的能力

​ nginx 通过 limit_req_zone 和 limit_req 两个配置来设定访问频率限制

Nginx 与 uWSGI

​ uWSGI 是 Python Web 应用程序的服务器，Python 的应用程序例如 Django 和 Flask 都是遵循的 uwsgi 协议，所以 nginx 与 python 的 Web 应用程序通讯，中间还需要通过 uWSGI 服务器

​ 一般代理使用的是 proxy_pass 的配置项来设置，将普通的 http 协议请求转发到 upstream 上游服务器，当要与 uWSGI 服务器通信时，则需要使用 uwsgi_pass 来设置，同时需要 include nginx 目录下的 uwsgi_params

​ 如果 Django 部署的时候，使用的不是 uwsgi，而是 gunicorn 运行服务，也无需使用 uwsgi_pass

​ 如果 nginx 和 uWSGI 是在一个服务器上安装的，可以不使用负载均衡，使用 unix socket 通信的速度会更快些
