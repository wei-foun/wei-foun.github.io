负载均衡（Load Balancing）

​ 软件层面负载均衡的方式大致可以分为以下几种类型：

​ http 重定向负载均衡：通过 http 302 重定向实现，处理性能方面要弱很多，是非常早期的负载均衡

​ dns 负载均衡：通过 dns 域名解析和缓存可以提升一定的处理性能，普通方式依然会暴露出应用服务器的 ip，一般场景下 dns 服务器并不直接指向应用服务器，而是中间再加一层负载均衡服务器，因此对一些大型网站域名发送 ping 时会可能出现不同的 ip

​ 反向代理负载均衡：典型的就是 nginx，静态请求可以直接由反向代理服务器来直接返回，动态请求从则可以通过反向代理服务器做代理请求，然后再将响应通过自己返回给客户端。反向代理工作在应用层 http 协议上，属于 7 层，适合中小型网络架构

​ ip 负载均衡：是工作在 tcp/ip 协议的网络层负载均衡（4 层），工作方式是会在收到客户端请求后，通过负载均衡得到应用服务器的 ip 后，将 ip 数据包中目标 ip 改为应用服务器 ip，将源 ip 改为自己，等应用服务器返回响应后，则会反向操作修改目标和源 ip 在返回到客户端

​ 这些操作可以在操作系统内核中完成，所以比起应用层的负载均衡效率要高，因为不需要建立握手，但响应数据如果较大，ip 负载均衡服务器就会一定程度上出现响应数据的流量瓶颈

​ 数据链路层负载均衡：是直接更改数据链路层头部的 mac 地址实现，这里应用服务器和负载均衡服务器会使用相同的虚拟 ip，这样 ip 路由就不会有修改，比起 ip 负载均衡可以处理更大量的数据响应

总结：

  • ip 和 数据链路层负载均衡 使用到的是 LVS（Linux Virtual Server）技术，性能可以达到十万级别
  • nginx 反向代理负载均衡处理性能可以达到 5 万左右
  • 除了以上这些，还有硬件层面的负载均衡设备，这种价格高昂，但处理性能也能达到百万级，同时也会有很多的功能

LVS

​ 是在 Linux 内核实现的高性能、高可用的负载均衡服务器（LVS，Linux Virtual Server），工作在 4 层网络中，只用于分发作用，所以资源消耗较低，性能相比 nginx 要高，但可配置和支持的功能相对而言就没有那么多，因此相对稳定不容易出错

​ LVS（ipvs）有三种负载均衡的模式，分别是： NAT，DR ，TUN

​ 一个常规的应用网络架构如下：nginx 前的 LB 服务器提供四层 ip 的负载均衡，内部网络通过 nginx 来提供内部域名的七层负载均衡

IPVS 工作模式

​ IPVS（IP Virtual Server）是工作在 Linux 内核态的 4 层负载均衡，主要有三种工作模式，分别是： NAT 模式、TUN 模式和 DR 模式

  1. NAT 模式
    1. 工作原理：客户端请求到达负载均衡器（LB）后，LB 根据负载均衡算法修改数据包的目标 ip 地址为某个真实服务器（RS）ip，并将请求转发给该真实服务器。真实服务器处理完请求后，响应数据发送回 LB，负载均衡器再修改数据包的源地址为虚拟服务地址（VIP），然后响应客户端，就是完成了一次 DNAT 和 SNAT 的转换
    2. 特点：适用于客户端和真实服务器不在同一网络的情况，且允许负载均衡服务器端口与真实服务的端口可以不一致，但是 LB 与 RS 必须在同一子网，并将 RS 的默认网关设置为 LB 的 vip
  2. TUN 模式（IP 隧道模式）
    1. 工作原理：LB 会将客户端的数据包做二次封装，会加上一个 ip 头其中 sip 是自己的 vip，dip 是 rs 的 ip，依次通过 IP 隧道发给真实服务器。真实服务器收到请求后，可以读到源报文和 LB 加的 ip 头部，然后将数据包的 ip 头里 sip 设置为 vip，dip 设置为客户端 ip
    2. 特点：允许 LB 和 RS 不必在同一子网内，且 LB 不负责响应，能减少 LB 的负载，适合大规模集群环境，通过 ip 隧道可以实现跨物理可用区通信
  3. DR 模式（直接路由模式）
    1. 工作原理：负载均衡器通过修改数据包的目标 MAC 地址，将请求负载到真实服务器。真实服务器处理完请求后，直接将响应数据发送回客户端而不再经过 LB
    2. 特点：性能最好，但它要求 LB 和 RS 必须在同一子网，RS 的 lo0 网卡需要绑定 LB 的 vip，且 LB 与 RS 的端口需要一致 ，以便通过链路层进行数据转发

补充：

​ 在 NAT 模式下，还延伸出一种 FullNAT 模式，这种模式相比 NAT 模式，会在 LB 处同时修改目标 ip 和源 ip 地址，这样可以实现 LB 与 RS 可以不用在同一子网内，这种模式下就允许部署多台 LB 来负载均衡流量，实现 LB 的高可用

​ 在 Kubernetes 中，kube-proxy 使用 IPVS 模式时，通常基于 NAT 模式工作

​ 更多参考：*https://mp.weixin.qq.com/s/9hmQluSpwDEn_gIlgM6c4w

ALB, NLB, CLB

​ 在一些公有云上，如 aliyun、aws 等，会提供专门的负载均衡配置管理服务，按照层级和功能定位会分为：CLB（传统负载均衡）、NLB（网络负载均衡）、ALB（应用负载均衡）

​ ALB 工作在 http/https 的七层负载均衡，支持通过 url 路径或域名 domain 来实现不同的转发策略，通常作为 api 网关流量入口，广泛使用在微服务，serverless 上

​ NLB 工作 tcp/udp 的四层传输层，拥有高性能、低延迟的特点，适合大规模流量的高效分发。通过 源 IP、源端口、目标 IP、目标端口、协议类型，来实现转发

​ CLB 定位通用型，可以支持四层和七层协议，但只支持基础的流量分发，不能根据特定需求不同的转发策略，适用于基础门户等单体的 web 应用

​ 在实际私有或公有云中，往往是组合方式使用服务负载均衡器 SLB（server load balancer），来实现兼顾性能，灵活等功能
