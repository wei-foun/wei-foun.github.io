docker 容器网络

发表于2023-07-05|更新于2025-08-09|container network
|阅读量:

容器网络基本概述

​ 对于 linux 容器来说它的网络栈实际是被一个 network namespace 所包住的，通过名称空间实现和宿主机的网络栈隔离，其中包括网卡，回环设备，路由表和 iptables 规则

​ 不过也可以使用 -net=host 的参数启动容器，这样这个容器就会直接使用宿主机的网络栈，容器内监听的端口就成了宿主机的实际端口

​ 这种方式虽然提供很好的网络性能，但容易出现端口冲突（宿主机端口永远是最稀缺的资源）

​ 因此常用的网络模式是利用网桥作为虚拟交换机来实现多容器间通信，因此如果使用宿主机网络栈，那么就必须提前规划好容器使用端口

​ 对于一个容器来说，它和宿主机之间通过一个虚拟设备 Veth Pair 进行连接，其中一端插在容器内部网络栈的网卡上（eth0），另一端则是在宿主机上（vethxxxx），所以 Veth Pair 可以将宿主机的 host namespace 与容器的 network namespace 连接

​ 在宿主机上，通过 brctl show 命令可以查看到在 docker0 上有一个 veth75c08a3 设备被插入，这个 docker0 网卡在安装 docker 时就会被自动安装

​ 当宿主机上的两个容器要进行通信时，数据包就会流入到 docker0 网桥上，而对于 docker0 对应的两个容器虚拟设备来说，他们将不会拥有数据包的处理权限，可以类比成交换机上的网口

​ 当容器一通信前，会发送 ARP 请求给到网桥，网桥通过 ARP 广播转发到其他的网口上，容器二收到后会将自己 eth0 网卡的 MAC 地址返回，此时容器一的数据包就明确了数据包的方向。数据包发送会同样进入到 docker0 网桥，网桥通过一个 MAC 地址映射表，将容器一数据包的目标 MAC 地址进行匹配，匹配到容器二的网口并将数据包发送过去

​ 同样，宿主机通过 ping 命令访问容器一时，请求一样会进入 docker0 网桥，根据映射规则将请求转发给对应的容器一的网口

​ 当容器一想要访问到另一个宿主机时，请求会从网桥来到宿主机所在网络，通过宿主机网络找到指定的宿主机，将数据包发送过去

​ 而如果容器一要访问集群中另一个宿主机上的容器时，这时候就在网桥上进一步衍生出了 overlay network 覆盖网络，可以将其看作是一个集群公用的一个特殊网桥

​ 不过其实不用这个覆盖网络，仅仅靠配置路由表一样可以实现容器访问不同节点的容器

容器网络 6 种模式

​ 包括 bridge、host、none、overlay、macvlan 和 ipvlan 六种模式

bridge

​ 默认的网络模式，利用 docker0 虚拟网桥连接宿主机和容器网络，容器会拥有独立的 ip。适用于单机容器网络，多容器可以用端口映射来提供访问

host

​ 是直接使用宿主机网络栈，容器不会拥有独立的 ip，端口也是直接使用宿主机的，但是性能接近原生。适用于高性能需求的容器通信方式

none

​ 不为容器创建任何网络环境，与宿主机网络完全隔离，容器内只会保留一个回环接口，这种模式下完全交由管理者来根据自己的需求来进行容器的网络配置

container

​ 共享之前创建已创建容器的网络，新创建的容器一样不会有自己的 ip，且端口也是共享的

macvaln

​ 这种模式下主机上多个虚拟网络接口设备有着不同的 mac 地址，所以允许 容器拥有独立的 mac 和 ip 地址，让容器像是虚机一样直接通过物理网络通信，实现需要容器应用与非容器应用或虚机混合的访问场景，属于 underlay 网络

ipvlan

​ 和 maclvan 一样支持直接使用物理网络，只是主机的多个虚拟网口是共享 mac 地址，这样可以减少 mac 地址消耗，也是属于 underlay 网络。使用 dhcp 分配 ip 时，客户端需要配置唯一标识告知 dhcp 服务器

overlay

​ 允许跨多主机的分布式网络通信，是在物理网络上建立的一层抽象网络覆盖多主机通信，并且内置了 dns 服务实现容器跨主机访问。通过 overlay 组网，容器跨主机通信时，物理网络只承载主机的网络通信，容器通信将在 overlay 的虚拟网络中进行

​ 其逻辑就是对网络数据包进行进一步封装，当容器要访问另一个网络的容器时，数据包传给所在宿主机后，宿主机会对此进行封住将自己的地址作为原地址，将目标容器所在宿主机地址作为目的地址，然后发送过去，目标容器宿主机进行解包，将数据包转发到自己内部网络的容器中，这个就是 VXLAN

​ overlay 网络架构是在跨主机，跨数据中心，以及多云(混合云) 场景下广泛使用的

VxLAN（virtual extensible lan）

​ 是常见的 overlay 网络（三层网络扩展二层网络）组建的一种技术，通过虚拟隧道端点设备（virtual tunnel endpoint VETP）对三层 UDP 数据包的进行二层 mac 地址二次封装实现跨节点通信（注意这个跨节点是三层实现的），从而不依赖底层的物理网络拓扑

​ vxlan 的典型场景：

  • 虚机或容器跨节点迁移，因为只要本身只要 ip 网络层可通，虚机 ip 就不用修改，依然可以通过 overlay 的二层进行跨节点通信
  • 用于解决 vlan 不足的问题，传统 vlan 最多可以创建 4094 个，但是 vlxan 可以创建 1600 万个虚拟网络

​ 但是 vxlan 也有局限性，因为扩展的虚拟隧道，涉及到数据包的封装和解封，在高流量情况下会增加 cpu 的负载

工作原理

  1. 服务器1 上的 10.0.0.1 容器向 服务器2 的 10.0.0.2 容器进行访问，ip 数据包经过主机的容器网桥后，发送到主机上的 VTEP 网络设备上
  2. 服务器1 的 VTEP 从数据包里获取目标容器的 mac 地址，然后在本机的路由表中查看 mac 地址对应的主机 ip
  3. 服务器1 的 VTEP 开始对数据包进行封装，先加上自己所在的虚拟网络标识（VNI，vxlan network identifier）作为 vxlan 头部，然后添加 UDP 头部来进行三层通信，再添加自己主机的真实 ip 和 mac 头部用来在网络中进行寻址转发
  4. VTEP 封装好数据包，将数据包通过服务器1 的物理网口发送，数据包在三层网络路由中会被确定数据包的转发
  5. 数据包最后被转发到服务器2 的网口上，服务器2 的 VTEP 收到 vxlan 的数据包，开始拆分这个 UDP 数据包
  6. 最后查看到数据包的 VNI 标识，确定数据包是要发送到 10.0.0.2 的虚拟地址，然后转发过去

IPIP（ip in ip）

​ 是 linux 原生内核支持的隧道 overlay 网络模式，不同于 vxlan 是通过三层扩展二层的 mac 地址封装，而是将二层的 mac 头部出掉，将源和目标主机的 ip 封装到新的 ip 数据包上实现跨节点通信

​ 相对 vxlan 封装的报文头部也更小，因为只多增加一个 ip 头部。但是 ipip 模式不支持组播和广播，可用于点对点的跨节点通信
