k8s-发布部署

发表于2025-08-03|更新于2025-08-03|kuberbetes
|阅读量:

部署策略

重建发布（recreate）

  • 优点：可以 一次性将所有 pod 重建，比如在测试中同一服务的 pod 分散在不同 node，想要快速更新时就可以使用这种方式
  • 缺点：升级过程中，服务将无法访问，会造成业务中断

滚动更新（rolling-update）

  • 优点：逐台滚动升级，例如三台设备的 v1 pod 升级，会先一台上创建 v2 pod，然后 v1 pod -= 1，v2 pod += 1，需要冗余设备协助升级，这个可以根据需要调整
  • 缺点：v1 到 v2 的升级过程中，流量是同时接入到 v1 与 v2 两个版本，会导致问题定位可能要复杂一点

蓝绿发布（blue-green）

  • 优点：升级切换和回退速度会更快，因为新旧版本的 pod 都是运行的，整个过程中保留所有旧 pod
  • 缺点：需要更多资源冗余，三台 v1 升级，需要创建三台 v2 全部完成后，将通过 service 将流量一刀切到 v2；故障影响范围大，v2 版本有 bug，可能会导致整个服务不可用

​ 使用蓝绿发布时，其实就是通过 label 来修改 svc，将 svc 的选择从篮色的 pod 改为绿色的 pod，所以蓝绿两个 deployment，上面的 yaml 中在 pod 的 label 上设置了 version 的标签，v1 表示篮色。在另外一样的 deployment 中，将 version 改为 v2 表示绿色，apply 后就会同时拥有蓝绿两个版本的 pod

​ 这个 svc 中，目前通过 selector 标签选择器，使用的是 v1 版本的绿色 pod，要改为蓝色的 v2 pod，只需将这里的 version 修改后重新 apply 即可

​ 注意，并不是一定需要按照这个实例使用 version 的标签来区分蓝绿，只要 pod 的 label 可以区分蓝绿让 svc 可以切换就行

金丝雀发布

  • 优点：在滚动发布之上，加入了流量控制实现灰度发布，当 v2 就绪后，会有小部分流量接入 v2，这样新版本出现故障后可以也能方便定位
  • 缺点：发布策略的复杂程度更高，整体的发布周期也更长

​ 金丝雀的部署配置和蓝绿的一模一样，只是对于金丝雀部署来说，因为允许流量同时接入 v1 和 v2，所以 service 的配置中 selector 没有了 version 字段。而对于流量控制，则需要配合 istio 等使用
